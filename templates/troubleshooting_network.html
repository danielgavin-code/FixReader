NET Troubleshooting
Network & Connectivity
Diagnosing FIX TCP connection failures, logon timeouts, firewall issues, TLS errors, and latency problems. Start with the quick triage checklist, then work through the relevant sections.
Quick Triage Checklist

Work through these in order before diving deeper. Most connectivity failures resolve at step 1, 2, or 4.

  • 1 Ping the host — confirms basic IP reachability. If ping fails, the problem is at the network layer, not the FIX layer. Check routing and DNS resolution first.
  • 2 Telnet or netcat to the FIX port — telnet fix-host.exchange.com 9001 or nc -zv fix-host.exchange.com 9001. A blank connection means TCP is up. "Connection refused" means the port is closed or filtered.
  • 3 Check your firewall rules — outbound traffic to the exchange's FIX port must be allowed. Most exchanges use ports in the 7000–9999 range; confirm the exact port in your exchange spec.
  • 4 Verify SenderCompID / TargetCompID — wrong comp IDs cause the exchange to reject the Logon silently or with a Logout. Confirm the exact string values (case-sensitive) with your exchange onboarding contact.
  • 5 Check the BeginString — connecting with FIX.4.2 to a FIX.4.4 session will be rejected. Confirm the FIX version in your exchange spec and in your session config.
  • 6 Review your application logs for the raw Logon message — is the message actually being sent? If no outbound Logon appears in logs, the TCP connection may be failing silently before FIX begins.
  • 7 Check session hours — most exchanges restrict FIX session availability to trading hours (e.g., 06:00–22:00 ET). Connecting outside the window will be refused at the TCP or FIX layer.
  • 8 Confirm the environment — UAT and production use different hostnames, ports, and comp IDs. A common mistake is connecting a production system to a UAT endpoint (or vice versa).
TCP vs FIX Session Layer

FIX runs over a persistent TCP connection. Connectivity problems fall into two distinct layers — and knowing which layer is failing tells you where to look.

TCP Layer (below FIX)
  • Connection refused — port is closed
  • Connection timed out — host unreachable or firewall dropped
  • Connection reset (RST) — server closed unexpectedly
  • SSL/TLS handshake failure
  • DNS resolution failure
FIX Session Layer (above TCP)
  • Logon rejected — wrong comp IDs or credentials
  • Logout received immediately after Logon
  • Wrong BeginString
  • Heartbeat timeout → disconnect
  • Session not available (outside hours)
Key distinction: If telnet to the FIX port connects successfully but your application cannot establish a FIX session, the problem is in the FIX session layer — not the network. Look at Logon / Logout message content, not firewalls.
Common TCP/IP Errors

Each TCP error has a distinct meaning and a different resolution path.

Error Meaning What to check
Connection refused Target host is reachable, but nothing is listening on that port Confirm the port number in your exchange spec. The FIX gateway may not be running, or session hours may have ended.
Connection timed out SYN packet sent but no SYN-ACK received within the OS timeout (~75s) Firewall is dropping your packets silently. Check outbound allow rules for the exchange's IP range and port. Also confirm the hostname resolves to the correct IP.
Connection reset (RST) TCP connection was established, then the remote host sent a RST Exchange server is actively rejecting after accepting — often a TLS mismatch, IP whitelist failure, or the FIX gateway restarted mid-session.
ETIMEDOUT / read timeout TCP connection is up but no data received within the read timeout window Heartbeat interval may be misconfigured. Check that HeartBtInt (108) in your Logon matches what the exchange expects. Also check for packet loss mid-session.
ECONNRESET (mid-session) Exchange closed the connection unexpectedly after session was established Usually follows a Logout message — check logs. Common causes: missed heartbeat, bad sequence number, invalid message, session timeout.
Logon Sequence & Timeout

When your application opens a TCP connection, the FIX session protocol expects a Logon (35=A) to be sent immediately. Most exchanges enforce a tight window — often 10–30 seconds — before closing the connection.

Tag 8 (BeginString) FIX.4.4 must exactly match the session version
Tag 35 (MsgType) A Logon message type
Tag 49 (SenderCompID) YOUR_FIRM_ID your assigned ID, case-sensitive
Tag 56 (TargetCompID) EXCHANGE_ID exchange's comp ID, per spec
Tag 108 (HeartBtInt) 30 heartbeat interval in seconds — must match exchange config
Tag 98 (EncryptMethod) 0 most venues require 0 (None) even when using TLS
Tag 34 (MsgSeqNum) 1 Logon on a new session must start at 1 unless resetting
Logon timeout: If the TCP connection is accepted but the Logon is not sent within the exchange's window (often 10s), they will close the socket. This appears in your logs as a dropped connection with no response. Always send the Logon immediately on connect — do not wait for user input or initialization steps.
Heartbeat & TestRequest Diagnostics

Once a session is established, the FIX heartbeat mechanism serves as the connectivity health check. Understanding it lets you diagnose mid-session disconnects.

Message 35= Purpose & when sent
Heartbeat 0 Sent every HeartBtInt seconds when no other messages have been sent. Confirms the session is alive.
TestRequest 1 Sent when no message has been received from the counterparty for HeartBtInt + tolerance seconds. Expects a Heartbeat in response containing the same TestReqID (112).
Heartbeat (response) 0 Response to TestRequest. Contains TestReqID (112) echoing the value from the TestRequest. If this is not received, the session is considered dead and a Logout / disconnect follows.
Diagnosing with heartbeat logs: Search your logs for 35=1 (TestRequest) and note whether a matching 35=0 response follows within a few seconds. If TestRequests are being sent repeatedly with no response, the exchange end is not receiving your messages — check for network partitions, one-way packet loss, or a crashed gateway process on their side.
# Search for TestRequest/Heartbeat pairs in a FIX log grep -E "35=1|35=0" fix_session.log | tail -40 # Look for TestReqID values that get no matching Heartbeat response grep "112=" fix_session.log
Firewall & Port Configuration

FIX sessions use persistent long-lived TCP connections. Stateful firewalls that time out idle connections are a common source of mid-session drops, especially in low-volume environments.

Exchange / Venue Typical Port Range Notes
NYSE Pillar (equities) 9001–9010 Ports vary by session type (order, drop copy). Confirm in the NYSE Pillar Gateway spec.
NASDAQ 7000–7999 Separate ports for UAT and production. Some sessions use TLS on a different port.
Cboe BZX / EDGX 9400–9500 Titanium spec lists exact ports per market and session type.
IEX 9000+ Confirm per environment in the IEX FIX specification.
Stateful firewall timeout: If your firewall has a TCP idle timeout shorter than your heartbeat interval, it will silently drop the connection state — your packets will be discarded without a RST. Set the firewall idle timeout to at least HeartBtInt × 3, or enable TCP keepalives on the socket.

IP whitelisting: Most exchanges require your outbound IP addresses to be whitelisted on their firewall. Submit your NAT/egress IPs to the exchange during onboarding. If you rotate IPs (e.g., cloud environments with dynamic IPs), you will need to update the whitelist each time.

# Test connectivity without your FIX application nc -zv fix-prod.exchange.com 9001 # Trace the TCP path — identifies where packets stop traceroute -T -p 9001 fix-prod.exchange.com # Check if a connection can be established and stays open nc -v fix-prod.exchange.com 9001 # (should remain connected — type anything; a clean disconnect means TLS or auth rejected)
SSL / TLS Configuration

Many exchanges now require TLS for FIX sessions, especially in production. TLS failures at the handshake stage look identical to a TCP RST from the application's perspective — the connection is dropped before any FIX bytes are exchanged.

TLS Error Likely Cause
SSL handshake timed out Port expects plain TCP but your client is sending a TLS ClientHello. Or firewall is blocking. Try connecting without TLS first.
Certificate verify failed Exchange's server certificate is signed by a CA not in your trust store, or the hostname in the cert doesn't match what you're connecting to. Add the exchange's root CA to your trust store.
Handshake failure (alert 40) No shared cipher suite. Exchange may require TLS 1.2+ or a specific cipher. Check whether your client is offering compatible cipher suites.
No certificate / Client cert required Exchange requires mutual TLS (mTLS). You must present a client certificate. Obtain this from the exchange during onboarding.
Certificate expired Either your client cert or the exchange's server cert has expired. Check both. Exchange certs rotate on a schedule — subscribe to their infrastructure notices.
# Inspect the exchange's TLS certificate and check expiry openssl s_client -connect fix-prod.exchange.com:9001 2>/dev/null | openssl x509 -noout -dates # Test TLS handshake and show cipher negotiated openssl s_client -connect fix-prod.exchange.com:9001 \ -tls1_2 -cipher 'ECDHE-RSA-AES256-GCM-SHA384' # Test with a client certificate (mTLS) openssl s_client -connect fix-prod.exchange.com:9001 \ -cert client.crt -key client.key
Note on EncryptMethod (Tag 98): Even when using TLS, most exchanges expect 98=0 (None) in the Logon message. TLS operates at the transport layer — it's invisible to FIX. Setting 98=1 (DES) or other values will cause a Logon rejection.
Latency & Packet Loss

High latency or intermittent packet loss causes heartbeat misses and unexpected session drops, even when basic connectivity looks fine. Measure before adjusting heartbeat intervals.

# Measure round-trip latency to the FIX endpoint ping -c 100 fix-prod.exchange.com # Detect packet loss across multiple hops mtr --report --report-cycles 100 fix-prod.exchange.com # Sustained bandwidth test (requires iperf3 server on the other side) iperf3 -c fix-prod.exchange.com -t 30
Symptom Likely cause Resolution
Repeated TestRequests, then disconnect Packet loss or high latency causing heartbeat misses Increase HeartBtInt (108) to give more tolerance, or fix the underlying network issue
Messages received out of order in logs TCP reordering (should not affect delivery — TCP guarantees order) Usually a logging artifact. Check if timestamps on logs are from message receipt vs. application processing.
Latency spikes at market open/close Network congestion at peak traffic times Consider dedicated/leased line connectivity. Shared internet paths are not suitable for latency-sensitive FIX.
Periodic disconnects every ~60 minutes Stateful firewall or NAT device timing out idle connection Enable TCP keepalives, reduce HeartBtInt, or increase firewall idle timeout
UAT vs Production Differences

UAT and production environments almost always have different hostnames, ports, comp IDs, and sometimes different FIX versions. Misconfiguration when promoting from UAT to production is a frequent source of connectivity failures.

Parameter UAT Production
Hostname uat-fix.exchange.com fix.exchange.com
Port 9001 (separate range) 9001 (same port, different host)
SenderCompID FIRMID_UAT FIRMID
TargetCompID EXCHANGE_UAT EXCHANGE
TLS required Often optional Usually required
IP whitelist Broader (dev IPs allowed) Strict (only production IPs)
Common mistake: Shipping a UAT config to production with comp IDs like FIRMID_UAT or TARGET_TEST. The exchange will refuse the Logon — often with no error message, just a Logout or RST. Always diff your UAT and production config files before going live.
Session hours differ: UAT environments often have shorter or different session hours than production. Confirm with your exchange contact which hours the UAT FIX gateway is available.
Packet Capture with tcpdump / Wireshark

When application-level logs are inconclusive, a packet capture shows exactly what bytes are on the wire — including whether TCP packets are being sent, received, dropped, or retransmitted.

# Capture all traffic to/from the FIX port, save for Wireshark tcpdump -i eth0 -w fix_capture.pcap host fix-prod.exchange.com and port 9001 # Quick inline view (human-readable, no save) tcpdump -i eth0 -A host fix-prod.exchange.com and port 9001 # Capture only TCP handshake and FIN/RST packets (low volume) tcpdump -i eth0 'host fix-prod.exchange.com and port 9001 and (tcp[tcpflags] & (tcp-syn|tcp-fin|tcp-rst) != 0)'

What to look for in the capture:

  • ✓ SYN → SYN-ACK → ACK — three-way handshake completed. TCP connection is up.
  • ✗ SYN → (no reply) — firewall is dropping your SYN packets. The host is either unreachable or a firewall is silently filtering.
  • ✗ SYN → RST — port is closed on the target. Nothing is listening, or the host is actively rejecting the connection.
  • ! Retransmits (e.g., [TCP Retransmission]) — packets are being sent but acknowledgment is not arriving. Indicates packet loss in one direction.
  • ! RST after data — exchange is closing the connection after receiving your Logon. Check for TLS mismatch, rejected comp IDs, or IP whitelist failure.
TLS sessions: Captures of TLS-encrypted sessions will not show FIX message content — only the TCP stream is visible. To inspect FIX message content on TLS connections, enable FIX message logging in your application before encryption, or use your FIX engine's built-in session log.
