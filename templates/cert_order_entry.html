P1 Environment & Connectivity 13 tests ►
1.1 Environment Checklist Basic 15 min
Overview

Before any FIX messages are exchanged, confirm the test environment is correctly configured. Exchanges operate separate certification environments with different IP addresses, ports, and CompIDs from production.

Why This Matters

Connecting to the wrong environment is the most common Day 1 mistake. Sending orders to a production environment during certification has caused real incidents.

Checklist
  • Obtain certification environment IP and port from exchange connectivity team
  • Confirm SenderCompID assigned for certification (different from production)
  • Confirm TargetCompID for certification environment
  • Obtain certification credentials (Username Tag 553, Password Tag 554 if required)
  • Confirm FIX version required (most US equity venues: FIX 4.2)
  • Confirm HeartBtInt agreed value (typically 30 seconds)
  • Verify firewall rules allow outbound TCP to exchange IP:port
  • Test raw TCP connectivity before attempting FIX logon
1.2 Network Connectivity Basic 10 min
Overview

Verify raw TCP connectivity to the exchange certification environment before attempting FIX logon.

Why This Matters

A FIX session that fails to connect is often misdiagnosed as a configuration problem when it is actually a network problem.

Sample Commands
telnet {exchange-ip} {port}
nc -zv {exchange-ip} {port}

Expected result: TCP connection established. If connection is refused or times out, the problem is network-level — not FIX configuration.

1.3 Session Configuration Basic 20 min
Overview

Verify your FIX session configuration matches exchange requirements exactly.

Required Session Parameters
  • BeginString (Tag 8): Must match exchange specification. US equity venues: FIX.4.2
  • SenderCompID (Tag 49): Your assigned certification CompID
  • TargetCompID (Tag 56): Exchange certification target (e.g. XNYS for NYSE)
  • HeartBtInt (Tag 108): Agreed interval in seconds (typically 30)
  • EncryptMethod (Tag 98): Must be 0 — no encryption (TLS handled at transport layer)
  • ResetSeqNumFlag (Tag 141): Set to Y on initial logon

1.8 Sequence Numbers Basic 20 min
Overview

Verify your application manages sequence numbers correctly — sequential, starting at 1, incrementing on every message.

Rules
  • MsgSeqNum (Tag 34) must start at 1 on a fresh session
  • Must increment by 1 for every outbound message
  • If exchange sends a message with a sequence number higher than expected — you have missed messages, send ResendRequest
  • If exchange sends a message with a sequence number lower than expected — reject it (possible duplicate)
1.9 ResetSeqNumFlag Basic 15 min
Overview

Verify your application correctly handles ResetSeqNumFlag (Tag 141) = Y on Logon, resetting both sides to sequence number 1.

Client sends logon with reset
Exchange responds at sequence 1
1.10 Resend Request Intermediate 20 min
Overview

Verify your application can send a ResendRequest (35=2) when it detects a gap in inbound sequence numbers and process the retransmitted messages.

Scenario: Exchange sends message with sequence 10, but you expected sequence 8.

Client sends ResendRequest

Tag 7 (BeginSeqNo) = 8, Tag 16 (EndSeqNo) = 0 means "through the last message sent." Exchange responds with messages 8 and 9 marked PossDup=Y.

1.11 Gap Fill Intermediate 25 min
Overview

Verify your application handles SequenceReset-GapFill (35=4, GapFillFlag=Y) correctly when the exchange skips over administrative messages during resend.

Scenario: During resend, exchange sends SequenceReset-GapFill to skip over Heartbeats and other administrative messages it does not need to retransmit.

Exchange sends

Tag 123 (GapFillFlag) = Y, Tag 36 (NewSeqNo) = 10 means "skip messages 8 and 9, next expected is 10."

Expected result: Your application updates its expected inbound sequence to 10 without requesting retransmission of messages 8–9.

1.12 Daily Startup Basic 15 min
Overview

Verify your application correctly initializes a FIX session at the start of each trading day.

Daily Startup Sequence
  1. Establish TCP connection
  2. Send Logon with ResetSeqNumFlag=Y (MsgSeqNum=1)
  3. Receive Logon response (MsgSeqNum=1)
  4. Begin sending orders
1.13 Daily Shutdown Basic 15 min
Overview

Verify your application correctly terminates a FIX session at end of trading day.

Daily Shutdown Sequence
  1. Cancel all open orders
  2. Wait for cancellation confirmations
  3. Send Logout (35=5)
  4. Receive Logout acknowledgement
  5. Close TCP connection

ExecutionReport (Partial Fill)

ExecutionReport (Full Fill)

Key tags: OrdStatus=2, LeavesQty=0, CumQty=OrderQty.

P5 Reject Scenarios 7 tests ►
5.1 Unknown Symbol Basic 15 min
Overview

Send an order with an invalid or unrecognized symbol. Client sends order with Tag 55=INVALIDXYZ.

Exchange responds: BusinessMessageReject (35=j) or ExecutionReport with OrdStatus=8 (Rejected). Tag 58 (Text) will contain description: "Unknown symbol".

5.2 Invalid Price Basic 15 min
Overview

Send a Limit order with a price outside acceptable range. Exchange responds with Rejected ExecutionReport or BusinessMessageReject.

5.3 Invalid Side Basic 10 min
Overview

Send an order with an invalid Side value (e.g., Tag 54=9). Exchange responds: Reject (35=3) at session level or BusinessMessageReject (35=j).

5.4 Duplicate ClOrdID Intermediate 20 min
Overview

Send two orders with the same ClOrdID. Expected: Second order rejected with Text indicating duplicate ClOrdID.

5.5 Missing Required Tags Basic 15 min
Overview

Send a NewOrderSingle with a required tag omitted (e.g., missing Tag 55 Symbol). Exchange responds: Reject (35=3) with RefTagID pointing to the missing tag and SessionRejectReason=1 (Required tag missing).

5.6 Business Message Reject Intermediate 20 min
Overview

Verify your application handles BusinessMessageReject (35=j) for valid FIX messages that fail business validation.

Exchange sends

Tag 372 (RefMsgType) = D indicates the rejected message was a NewOrderSingle. Tag 380 (BusinessRejectReason) values: 0=Other, 1=Unknown ID, 2=Unknown Security, 3=Unsupported Message Type.

5.7 Risk Reject Advanced 25 min
Overview

Verify handling when an exchange risk control rejects an order — typically oversized quantity or excessive notional value.

P6 Advanced Scenarios 6 tests ►
6.1 Disconnect During Active Order Advanced 30 min
Overview

Simulate a network disconnect while an order is live at the exchange. Verify correct reconnect and state recovery behavior.

Procedure
  1. Send a Limit order
  2. Receive New acknowledgement
  3. Simulate disconnect (kill TCP connection)
  4. Reconnect and send Logon WITHOUT ResetSeqNumFlag=Y
  5. Send ResendRequest for any missed messages
  6. Verify order status
6.2 PossDup Processing Advanced 30 min
Overview

Verify your application correctly handles messages marked PossDup=Y (Tag 43). When the exchange retransmits a message it will set Tag 43=Y and Tag 122 (OrigSendingTime) to the original transmission time.

Your application must: accept the message if it has not been processed, discard it if already processed (same ExecID), never reject a PossDup message outright.

6.3 Duplicate ExecID Advanced 25 min
Overview

Verify your application handles the case where two ExecutionReports arrive with the same ExecID — one original, one retransmitted.

6.4 Gap Fill Recovery Advanced 30 min
Overview

Full recovery scenario — simulate missing a range of messages and verify complete gap fill processing.

Procedure
  1. Detect gap in inbound sequence numbers
  2. Send ResendRequest (35=2) for the missing range
  3. Process retransmitted messages (marked PossDup=Y)
  4. Process GapFill (35=4) for any skipped administrative messages
  5. Verify final state matches expected state
6.5 Late ExecutionReports Advanced 25 min
Overview

Verify your application handles ExecutionReports that arrive out of order or delayed.

6.6 Exchange Restart Advanced 30 min
Overview

Verify your application correctly recovers when the exchange system restarts during the trading session.

Typical sequence: Exchange sends Logout, closes connection, comes back up. Your application must reconnect with proper sequence number handling.

Lessons Learned
  • ›Sequence number discipline is the foundation of every FIX session
  • ›Always handle OrderCancelReject — never assume a cancel succeeded
  • ›Fill reconciliation: CumQty must always equal the sum of all LastQty values
  • ›PossDup processing is required, not optional
  • ›Always verify connectivity at TCP level before debugging FIX configuration
  • ›ExecID is your idempotency key for ExecutionReports
  • ›DoneForDay is not terminal for GTC orders — Expired is terminal
  • ›After any disconnect, do not assume order state — always query
