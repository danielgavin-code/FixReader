Gap Fill & Resend
Understand how FIX sessions recover missing messages, distinguish a valid Gap Fill from a dangerous reset, and diagnose resend loops.
Critical Warning
Do not use Sequence Reset to hide missing business messages.

A Gap Fill may safely skip messages that should not be replayed, but missing orders, executions, cancels, replaces, busts, or corrections still require reconciliation.

Before forcing sequence state forward:

Quick Triage
  • 1 Identify the missing sequence range. The sequence gap is the difference between the last sequence number received and the next expected sequence number.
  • 2 Find the Resend Request (35=2). Confirm which side sent it — the receiver of the gap sends the request to the sender who missed the messages.
  • 3 Check BeginSeqNo (7) and EndSeqNo (16). These define the requested range. EndSeqNo = 0 means "through the last message sent."
  • 4 Review whether messages were resent or gap-filled. Application messages should be resent with PossDupFlag (43) = Y. Session messages and non-replayable messages are gap-filled.
  • 5 Check GapFillFlag (123) and NewSeqNo (36). 123=Y confirms this is a Gap Fill, not a hard reset. NewSeqNo shows where the sequence will resume.
  • 6 Confirm the session returned to the expected sequence. After recovery, both sides must agree on the next expected inbound sequence number.
Core Messages
Message Purpose
Resend Request 35=2 Requests missing messages in a sequence range
Sequence Reset 35=4 with 123=Y Gap Fill — skips sequence numbers that will not be replayed
Sequence Reset 35=4 without 123=Y Hard reset or administrative sequence adjustment
Resent message with 43=Y Replayed business message — counterparty must deduplicate
Reading the Messages

Example 1 — Resend Request:

35=2|7=205|16=207|
  • • Requesting messages 205 through 207.
  • • The sender must replay recoverable application messages in that range, or gap-fill any that should not be resent.
  • • Heartbeats and Test Requests are never replayed — they are always gap-filled.

Example 2 — Gap Fill response:

35=4|123=Y|36=208|
  • • GapFillFlag (123) = Y confirms this is a Gap Fill, not a hard reset.
  • • NewSeqNo (36) = 208 advances the expected sequence to 208. Messages before 208 will not be replayed.
  • • Confirm what was in the skipped range before accepting this as a complete recovery.
Gap Fill vs Hard Reset
Gap Fill Hard Reset
Normal recovery tool Administrative action
Uses 123=Y Usually omits 123=Y
Preserves session continuity Changes expected state directly
Skips non-replayable messages May bypass unresolved history
Common during resend Rare and high-risk

A Gap Fill is part of recovery. A hard reset changes the session state. They are not interchangeable.

Common Problems
Symptom Likely Cause First Check
Endless Resend Requests Requested range never closes NewSeqNo (36) and whether it covers the full range
Same range requested repeatedly Gap Fill not accepted GapFillFlag (123) and NewSeqNo (36) values
Duplicate business messages Replayed messages treated as new PossDupFlag (43) handling in the application
Session jumps forward unexpectedly Hard reset used instead of Gap Fill Sequence Reset message — confirm 123=Y is present
Missing fills after recovery Business messages gap-filled incorrectly Execution reconciliation for the affected range
Resend starts but never completes Store or replay failure on the sender Sender message store and replay logs

5-Minute Diagnosis
Gap Fill & Resend — TL;DR
  1. What is the missing sequence range — BeginSeqNo to EndSeqNo?
  2. Were application messages in that range? If so, which ones?
  3. Was recovery handled with a Gap Fill (123=Y) or a hard reset?
  4. Why is the Resend Request looping — does NewSeqNo close the gap?
  5. Have orders and executions in the affected range been reconciled?
