Order Rejects
Diagnose why an order was rejected, identify whether the rejection came from the FIX session, broker, risk layer, or exchange, and determine the safest next step.
Critical Warning
Do not resend a rejected order until you understand why it failed.

Blindly resubmitting may create:

  • duplicate orders
  • repeated rejects
  • unintended exposure
  • broken ClOrdID chains
  • inconsistent order state
  • additional risk or compliance breaches

Before resending:

Quick Triage
  • 1 Identify the rejection message type. A session 35=3, a Business Message Reject 35=j, and an Execution Report 35=8 reject each point to a different layer.
  • 2 Capture the rejecting text and reason code. Read Text (58) and the applicable reason tag — 103, 371, or 373 — before doing anything else.
  • 3 Match the reject back to the original ClOrdID. Confirm exactly which outbound order was rejected and whether any order state was created at the counterparty.
  • 4 Determine where the rejection came from. FIX engine validation, broker risk, or exchange rules each require a different fix.
  • 5 Compare the rejected order with a known-good order. For the same venue, account, and symbol type — look for differences in required tags, enum values, or routing.
  • 6 Correct the cause before resubmitting. Use a new ClOrdID if the original was consumed, even partially.
First Question: What Rejected the Order?
Message Meaning Typical Source
Reject 35=3 Session-level FIX problem FIX engine
Business Message Reject 35=j Unsupported or invalid application message Counterparty FIX gateway
Execution Report 35=8, 150=8, 39=8 Order rejected Broker, risk layer, or exchange
Order Cancel Reject 35=9 Cancel or replace rejected Broker or exchange

The message type tells you which layer failed. Do not troubleshoot an exchange order reject the same way you troubleshoot a session Reject.

Key Reject Fields

Pull these fields from the reject message first. They identify what failed and where.

Tag Purpose
11ClOrdID — your original order identifier
37OrderID — exchange-assigned identifier, if any was created
17ExecID — unique execution identifier on the reject message
58Text — human-readable rejection reason; read this first
103OrdRejReason — coded reason for an Execution Report reject
371RefTagID — points to the specific tag that caused a session Reject
372RefMsgType — identifies the message type that was rejected
373SessionRejectReason — coded reason for a session-level Reject
380BusinessRejectReason — coded reason for a Business Message Reject
150ExecType — 8 confirms this Execution Report is a reject
39OrdStatus — 8 confirms the order is in a rejected state
Reading the Logs

Example 1 — Execution Report reject from the broker:

OUT 35=D|11=ABC123|55=AAPL|54=1|38=100|40=2|44=185.50| IN 35=8|11=ABC123|150=8|39=8|103=99|58=Account not authorized|
  • • The order reached the counterparty — a response was returned.
  • • It was rejected through an Execution Report — 150=8 and 39=8 confirm the order is in a rejected state.
  • • The ClOrdID matches the original outbound order.
  • • The cause is account authorization — not FIX syntax. Fix the account entitlement before resubmitting.

Example 2 — Session-level Reject pointing to a specific tag:

IN 35=3|371=54|372=D|373=5|58=Value is incorrect|
  • • The session rejected a New Order Single (372=D) before it reached the application layer.
  • • RefTagID (371) = 54 points directly to Side. Fix the malformed or unsupported Side value before resending.
Compare Against a Known-Good Order

When the rejection is unclear, compare the failed message with a recently accepted order for the same venue, account, symbol type, order type, TimeInForce, and session.

Look for differences in:

  • • required tags present vs. absent
  • • enum values — especially OrdType (40), Side (54), and TimeInForce (59)
  • • account or trader identifiers
  • • routing instructions
  • • price precision and tick size
  • • instrument identifiers — Symbol, SecurityID, and SecurityIDSource
Use Compare Messages to isolate the differences between the rejected order and a known-good order.

5-Minute Diagnosis
Order Rejects — TL;DR
  1. What message type was the reject — 35=3, 35=j, 35=8, or 35=9?
  2. What does Text (58) say? What do the reason codes point to?
  3. Does any order state exist at the counterparty?
  4. Is the cause a FIX field, an entitlement, or a risk rule?
  5. Is it safe to resend — and do you need a new ClOrdID?
