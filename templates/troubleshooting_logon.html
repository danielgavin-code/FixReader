Logon Failures
Diagnose why a FIX session is rejected during Logon (35=A), identify the rejection reason, and establish a healthy session.
Critical Warning
A successful TCP connection does not mean FIX Logon succeeded.
Quick Triage
  • 1 Did the TCP connection establish successfully? Use telnet fix-host port or nc -zv fix-host port. If TCP fails, the problem is at the network layer — not FIX.
  • 2 Was Logon (35=A) actually sent? Check your application's outbound message log. If no Logon appears, the issue is in your engine's startup sequence.
  • 3 Did the exchange respond with Logout (35=5) or Reject (35=3)? The Text (58) field almost always says why — read it before anything else.
  • 4 Verify SenderCompID (49), TargetCompID (56), and BeginString (8) — these three fields are case-sensitive and must match the exchange's records exactly.
  • 5 Verify credentials and session hours — Username (553) and Password (554) if required, and confirm the current time is within the exchange's session window.
  • 6 Review sequence numbers before attempting any reset — a sequence mismatch is common after a hard restart. Check both sides' expected values before touching ResetSeqNumFlag (141).
Common Logon Problems
Symptom Likely Cause Check

Required Logon Fields
Tag Purpose
8 BeginString — FIX version, must match exchange spec exactly
35=A Logon message type
34 MsgSeqNum — 1 for a new session; must match counterparty's expected value
49 SenderCompID — your assigned ID, case-sensitive, no trailing spaces
56 TargetCompID — exchange's comp ID, per spec
98 EncryptMethod — must be 0 (None) even when using TLS
108 HeartBtInt — heartbeat interval in seconds; must match exchange config
553 Username — required by some exchanges; confirm with onboarding documentation
554 Password — required by some exchanges; confirm with onboarding documentation
141 ResetSeqNumFlag — only include when performing a coordinated reset; omit otherwise
Reading the Logs

Example 1 — Logon rejected with invalid SenderCompID:

OUT 35=A|49=CLIENT|56=NYSE|34=1|98=0|108=30| IN 35=5|58=Invalid SenderCompID|
  • • TCP succeeded — the session reached the FIX layer and the Logon was received by the exchange.
  • • The exchange responded with a Logout — Tag 58 says exactly why: Invalid SenderCompID.
  • • The network is healthy; the configuration is not. Fix Tag 49 before trying again — do not reset sequence numbers.

Example 2 — No response after Logon:

OUT 35=A|49=CLIENT|56=NYSE|34=1|98=0|108=30| (no inbound response) TCP disconnect after ~10s

Likely causes:

  • • Firewall dropped the Logon before it reached the exchange — run tcpdump to confirm bytes were transmitted
  • • TLS handshake failure — the connection was dropped before FIX bytes could be exchanged
  • • Session unavailable — the exchange gateway is outside its session window and closed the connection on receipt
  • • IP not whitelisted — the exchange accepted the TCP connection then dropped it before processing the Logon
Before Contacting the Exchange

Collect these before opening a ticket or calling support. Most exchange support teams will ask for all of them immediately.

  • • SenderCompID used in the Logon
  • • TargetCompID used in the Logon
  • • BeginString (FIX version)
  • • MsgSeqNum sent in the Logon
  • • HeartBtInt value
  • • Username used (if applicable)
  • • Timestamp of the failed Logon attempt (UTC)
  • • Full outbound Logon message (raw FIX)
  • • Full inbound Logout or Reject response (raw FIX), if any
Recovery Checklist

Work through this before retrying the Logon. Retrying with the same misconfiguration produces the same result.

Logon Failures — Pre-Retry Checklist
5-Minute Diagnosis
Logon Failures — TL;DR
  1. Is TCP up? (telnet or nc to the FIX port)
  2. Did Logon go out? (check outbound application logs)
  3. What was the first response — Logout (35=5), Reject (35=3), or silence?
  4. Network, configuration, or sequence?
  5. Only reset after you've proven recovery isn't possible.
